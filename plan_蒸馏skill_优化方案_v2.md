## 三、质量优化：聚焦提取质量提升

### 3.1 提取质量评估维度

| 维度 | 评估标准 | 权重 |
|------|---------|------|
| **原文支撑度** | 每条规则有≥2处原文引用，引用精确≤150字 | 30% |
| **决策清晰度** | A2（触发场景）明确，E（执行步骤）具体 | 25% |
| **边界清晰度** | B（边界与盲点）≥1条，局限性明确 | 20% |
| **跨书一致性** | 规则在≥2本书中出现（单书例外除外） | 15% |
| **可执行性** | 规则可直接用于写作决策 | 10% |

### 3.2 提取质量提升措施

#### 3.2.1 强化原文引用要求

**现状**：部分规则原文引用不足
**改进**：
- 每条规则必须有≥2处原文引用
- 引用必须精确到章节号
- 引用长度≤150字，但必须包含关键决策点

#### 3.2.2 强化决策清晰度

**现状**：部分规则A2/E不够具体
**改进**：
- A2必须是「如果{条件}，则{行动}」格式
- E必须是1-3个具体步骤
- 禁止模糊表述（如"根据情况灵活把握"）

#### 3.2.3 强化边界清晰度

**现状**：部分规则缺少边界说明
**改进**：
- 每条规则必须有B（边界与盲点）
- B必须说明规则的局限性
- B必须说明不适用的场景

### 3.3 提取质量验证流程

```
Phase 2 提取 → Phase 3 验证 → Phase 3.5 质量评分 → Phase 4 构造
```

**Phase 3.5 质量评分**：
- 对每条规则按5个维度评分
- 总分≥4.0：高质量，直接写入
- 总分3.0-3.9：中等质量，写入但标注「需验证」
- 总分<3.0：低质量，写入rejected/

## 五、缓存机制设计

### 5.1 缓存目标

**目标**：减少重复计算，提升蒸馏效率
**范围**：Phase 1（整书理解）和 Phase 2（并行提取）

### 5.2 缓存策略

#### 5.2.1 缓存层次

| 层次 | 缓存内容 | 缓存位置 | 失效条件 |
|------|---------|---------|---------|
| **L1** | 原文哈希 | `.distill/cache/hash.json` | 原文文件变化 |
| **L2** | 整书理解 | `.distill/cache/book-overviews/` | 原文哈希变化 |
| **L3** | 提取结果 | `.distill/cache/candidates/` | 整书理解变化或提取器prompt变化 |

#### 5.2.2 缓存键设计

```json
{
  "cache_key": {
    "book_hash": "sha256:abc123...",
    "extractor_version": "v2.1",
    "prompt_hash": "sha256:def456...",
    "timestamp": "2025-06-02T10:00:00Z"
  }
}
```

#### 5.2.3 缓存失效机制

**自动失效**：
- 原文文件变化 → L1缓存失效 → L2/L3缓存失效
- 提取器prompt变化 → L3缓存失效
- 整书理解变化 → L3缓存失效

**手动失效**：
- 用户可手动清除缓存：`/story-distill --clear-cache`
- 用户可强制重新提取：`/story-distill --no-cache`

#### 5.2.4 缓存清理策略

**定期清理**：
- 缓存超过7天自动清理
- 缓存大小超过100MB自动清理最旧缓存

**空间管理**：
- 缓存目录：`.distill/cache/`
- 最大缓存大小：100MB
- 缓存清理日志：`.distill/cache/cleanup.log`

### 5.3 缓存实现

#### 5.3.1 缓存检查流程

```
Phase 0: 检查缓存
  ├── 计算原文哈希
  ├── 检查L1缓存是否存在
  │   ├── 存在 → 使用缓存
  │   └── 不存在 → 继续Phase 1
  └── 检查L2缓存是否存在
      ├── 存在 → 使用缓存
      └── 不存在 → 继续Phase 1
```

#### 5.3.2 缓存写入流程

```
Phase 1完成: 写入L2缓存
  ├── 保存整书理解结果
  ├── 计算缓存键
  └── 写入缓存目录

Phase 2完成: 写入L3缓存
  ├── 保存提取结果
  ├── 计算缓存键
  └── 写入缓存目录
```

#### 5.3.3 缓存读取流程

```
Phase 0: 读取缓存
  ├── 计算原文哈希
  ├── 查找匹配的缓存键
  ├── 验证缓存完整性
  └── 返回缓存结果
```

### 5.4 缓存风险控制

#### 5.4.1 缓存一致性风险

**风险**：缓存与实际数据不一致
**控制**：
- 每次读取缓存时验证哈希
- 缓存写入时记录时间戳
- 缓存读取时检查时间戳是否过期

#### 5.4.2 缓存空间风险

**风险**：缓存占用过多空间
**控制**：
- 设置最大缓存大小（100MB）
- 定期清理过期缓存
- 缓存清理时记录日志

#### 5.4.3 缓存性能风险

**风险**：缓存读取反而降低性能
**控制**：
- 缓存读取前检查文件大小
- 小文件直接读取，不缓存
- 缓存命中率监控
