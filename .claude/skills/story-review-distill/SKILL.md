# story-review-distill：审稿蒸馏

**核心理念：好编辑的能力不是「挑毛病」，是「给方案」。我们要提取的是她怎么判断、怎么反馈、怎么改稿。**

**与 story-distill 的关系**：
- story-distill：从作者的小说中提取「怎么写」→ 写作决策框架
- story-review-distill：从编辑的反馈中提取「怎么改」→ 审稿技法框架
- 两者互补：distill 产出写作 SKILL，review-distill 产出 review SKILL 的审稿技法 section

---

## 输入要求

| 项目 | 要求 |
|------|------|
| 编辑反馈 | 3-5 份真实编辑审稿意见（文本/截图/文档） |
| 格式 | 每份反馈需包含：原稿片段 + 编辑意见 + 修改建议 |
| 来源 | 真实编辑（如美蛙小编）、AI审稿、自我审稿 |

**如果只有 1 份反馈**：正常继续，跳过跨反馈验证。

---

## Phase 0：输入验证

1. 检查反馈文本是否存在、格式是否正确
2. 创建输出目录结构：
   ```
   .claude/skills/story-style/{作者名}/review/
   ├── references/
   │   ├── editor-feedback/     # 原始反馈文本
   │   ├── feedback-analysis/   # 反馈分析
   │   └── candidates/          # 提取的审稿技法
   └── SKILL.md                 # 最终产出
   ```
3. 复制反馈文本到 `editor-feedback/`

---

## Phase 1：反馈精读

**并行** spawn N 个 Task sub-agents（每份反馈一个）：

每个 agent 负责一份反馈的：
1. 反馈结构分析（表扬→问题→建议 的比例和顺序）
2. 问题类型分类（结构/人设/节奏/文笔/商业）
3. 反馈语气分析（严厉/温和/鼓励/专业）
4. 修改建议提取（给的具体建议是什么）
5. 重写示范提取（编辑是否给了改后的文字）

**产出**：每份反馈写入 `references/feedback-analysis/{反馈编号}.md`

---

## Phase 2：并行提取

**并行** spawn 4 个 Task sub-agents：

| subagent | 提取什么 | 产出文件 |
|----------|---------|---------|
| 1 审稿话术 | 编辑怎么表扬、怎么批评、怎么给建议 | `candidates/review-voice.md` |
| 2 问题分类树 | 编辑关注什么问题、怎么分类、严重等级 | `candidates/problem-taxonomy.md` |
| 3 重写技法 | 编辑给的改前改后示例，提取改写规律 | `candidates/rewrite-techniques.md` |
| 4 评分维度 | 编辑关注什么维度、权重多大 | `candidates/editor-scoring.md` |

### 提取器 1：审稿话术

从编辑反馈中提取**表达方式**：

| 维度 | 找什么 |
|------|--------|
| **表扬方式** | 编辑怎么夸？夸什么？用什么句式？ |
| **批评方式** | 编辑怎么指出问题？直接说还是委婉说？ |
| **建议方式** | 编辑怎么给建议？给方向还是给具体步骤？ |
| **语气特征** | 专业？亲切？严厉？幽默？ |

**产出格式**：
```markdown
## 审稿话术

### 表扬模板
- 「{亮点描述}非常带感/很聪明/执行到位」
- 「你成功解决了{问题}」

### 批评模板
- 「{问题描述}，这是{严重程度}的问题」
- 「目前的处理会导致{后果}」

### 建议模板
- 「建议在{位置}增加{内容}」
- 「参考{对标作品}的处理方式」
```

### 提取器 2：问题分类树

从编辑反馈中提取**问题分类体系**：

| 维度 | 找什么 |
|------|--------|
| **结构问题** | 视角偏移、情节逻辑、节奏失控 |
| **人设问题** | 角色撞型、人设崩塌、动机单薄 |
| **文笔问题** | AI痕迹、对话机器感、动作流水账 |
| **商业问题** | 平台不适配、钩子不够、爽点不足 |

**产出格式**：
```markdown
## 问题分类树

### 🔴 致命问题（直接打回）
- 主角视角偏移（男二喧宾夺主）
- 核心逻辑漏洞（角色行为不合理）
- 人设撞型（两个男主太像）

### 🟡 重要问题（需要修改）
- AI痕迹明显（动作流水账、对话机器感）
- 爽点不足（钩子不够强）
- 平台不适配（番茄要爽、晋江要性张力）

### 🟢 优化问题（可改可不改）
- 时代细节不足
- 外貌描写模板化
```

### 提取器 3：重写技法

从编辑给的改前改后示例中提取**改写规律**：

| 维度 | 找什么 |
|------|--------|
| **动作描写升级** | 怎么把「流水账动作」改成「电影感动作」？ |
| **对话润色** | 怎么把「机器感对话」改成「有人味的对话」？ |
| **环境描写增强** | 怎么把「功能性环境」改成「情绪性环境」？ |
| **逻辑修补** | 怎么修补角色行为的逻辑漏洞？ |

**产出格式**：
```markdown
## 重写技法

### 动作描写升级
**改写规律**：在动作中加入微表情+环境光影+节奏控制
**改前**：他站直了，整了整衬衫，把纽扣扣上
**改后**：他慢条斯理将卷起的袖口一寸寸放下，带血的手指扣上领口最顶端的纽扣
**为什么更好**：有画面感、有细节、有节奏

### 对话润色
**改写规律**：删掉冗余标签，让对话内容本身传递语气
**改前**：「你来了。」她轻声说道
**改后**：「你来了。」她抬眼看向门口
**为什么更好**：用动作代替语气标签，更有画面感
```

### 提取器 4：评分维度

从编辑反馈中提取**评分标准**：

| 维度 | 找什么 |
|------|--------|
| **故事维度** | 编辑关注情节的什么方面？ |
| **人设维度** | 编辑关注角色的什么方面？ |
| **文笔维度** | 编辑关注文字的什么方面？ |
| **商业维度** | 编辑关注市场的什么方面？ |

**产出格式**：
```markdown
## 编辑评分维度

| 维度 | 权重 | 编辑关注什么 |
|------|------|------------|
| 故事 | 35% | 情节逻辑、节奏控制、冲突密度 |
| 人设 | 30% | 角色差异化、动机清晰度、弧光完整度 |
| 文笔 | 20% | AI痕迹、对话自然度、动作描写质量 |
| 商业 | 15% | 钩子强度、平台适配、爽点密度 |
```

---

## Phase 3：验证

**并行** spawn 2 个 Task sub-agents：

| subagent | 验证什么 | 产出 |
|----------|---------|------|
| 1 跨反馈验证 | 同一技法在 ≥2 份反馈中出现 | 标记「可能偶然」的技法 |
| 2 可执行性验证 | 每个技法是否有具体的改写步骤 | 不可执行的写入 `rejected/` |

---

## Phase 4：合成输出

将验证通过的审稿技法写入 review SKILL.md 的「审稿技法」section：

```markdown
## 审稿技法

### 审稿话术
{从 review-voice.md 提取}

### 问题诊断
{从 problem-taxonomy.md 提取}

### 重写示范
{从 rewrite-techniques.md 提取}

### 编辑评分
{从 editor-scoring.md 提取}
```

**产出路径**：`.claude/skills/story-style/{作者名}/review/SKILL.md`（追加到已有 review SKILL.md 的末尾）

---

## 与 story-distill 的协作

```
story-distill（作者蒸馏）
├── Phase 1-5：提取「作者怎么写」
├── 产出：写作 SKILL.md + review/SKILL.md（红线+处方+阈值）
└── review/SKILL.md 缺少「审稿技法」section

story-review-distill（审稿蒸馏）
├── Phase 1-4：提取「编辑怎么改」
├── 产出：review/SKILL.md 的「审稿技法」section
└── 追加到 story-distill 产出的 review/SKILL.md 末尾
```

**执行顺序**：
1. 先跑 story-distill --mode=review → 产出 review/SKILL.md（红线+处方+阈值）
2. 再跑 story-review-distill → 追加「审稿技法」section
3. 最终 review/SKILL.md = 红线+处方+阈值+审稿技法

## 反例黑名单

| # | 反模式 | 替代做法 |
|---|--------|---------|
| 1 | 把通用审稿建议当技法（「写得更好」） | 每个技法必须有改前改后示例 |
| 2 | 只提取批评不提取表扬 | 话术必须包含表扬+批评+建议 |
| 3 | 忽略编辑的语气特征 | 语气是话术的核心，必须提取 |
| 4 | 问题分类太粗（「文笔不好」） | 分类必须到可操作的粒度 |
