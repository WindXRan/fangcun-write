## 二、两个 cangjie 提取器重新评估

### 2.1 信念形成故事提取器

**现状分析**：
- `mental-model-extractor.md` 已提取「故事观」「角色观」「冲突观」「爽感观」
- 这些本质上就是作者的「信念」——他们相信什么是好故事

**结论**：**P2 可选，不单独建提取器**

**替代方案**：
- 在 `mental-model-extractor.md` 中增加一个维度：「信念形成路径」
- 要求提取：作者的信念是如何从具体案例中形成的（故事→信念的推导链）

```markdown
## 增加到 mental-model-extractor.md

### 信念形成路径（可选维度）

每个心智模型可额外标注：
- **形成推断**：从哪些具体选择推断出这个信念
- **反例检验**：有没有违背这个信念的案例

示例：
- 信念：好故事 = 读者和主角一起醒来
- 形成推断：6本书的第1章都用「信息盲区」开场
- 反例检验：《XX》第1章用了旁白开场，但那是回忆杀，不违背信念
```

### 2.2 真实决策案例提取器

**现状分析**：
- `decision-heuristic-extractor.md` 已提取「如果X，则Y」的决策规则
- 每条规则都有「A1（书中案例）」——这就是决策案例

**结论**：**P2 可选，不单独建提取器**

**替代方案**：
- 在 `decision-heuristic-extractor.md` 中强化「案例深度」
- 要求：每个案例不只是「第X章用了这个规则」，而是「在X场景，作者选择了Y，效果是Z」

```markdown
## 强化 decision-heuristic-extractor.md 的案例要求

### A1 案例深度要求

原要求：至少1个具体应用（如「《惜花芷》第1章」）

新要求：至少1个**深度案例**，包含：
- **场景**：什么情况下（如「开篇需要快速代入」）
- **选择**：作者做了什么（如「用他人对话碎片开场」）
- **效果**：达到什么效果（如「读者和主角一起困惑，代入感极强」）
- **反事实**：如果不这样做会怎样（如「用旁白交代背景，代入感弱」）
```

## 三、补充缺失的能力

### 3.1 规则质量量化（P1）

**目的**：给每条规则打分，筛选高质量规则

**方案**：在 Phase 3 三重验证后，增加「质量评分」

```markdown
## Phase 3.5：规则质量评分

每条通过三重验证的规则，按以下维度评分（1-5分）：

| 维度 | 评分标准 | 权重 |
|------|---------|------|
| **原文支撑度** | 有几处原文引用？引用是否精确？ | 30% |
| **跨书一致性** | 在几本书中出现？频率如何？ | 25% |
| **可执行性** | A2 是否清晰？E 是否具体？ | 25% |
| **独特性** | 与通用规则差异多大？ | 20% |

**总分 = 加权平均**

**筛选阈值**：
- ≥4.0：高质量，直接写入 SKILL.md
- 3.0-3.9：中等质量，写入但标注「需验证」
- <3.0：低质量，写入 rejected/

**输出**：每条规则附质量分数，写入 `references/candidates/` 对应文件
```

### 3.2 整体框架评估（P1）

**目的**：评估蒸馏出的整体框架是否自洽、可用

**方案**：Phase 7 增加「框架级 judge」

```markdown
## Phase 7.3：框架级 Judge 评估

**独立 judge agent** 读取完整的 SKILL.md，评估整体框架：

### 评估维度

| 维度 | 问题 | 评分 |
|------|------|------|
| **内在一致性** | 心智模型之间是否矛盾？决策规则是否冲突？ | 1-5 |
| **覆盖完整性** | 开篇→中段→结局是否都有决策规则？ | 1-5 |
| **可组合性** | 规则之间能否组合使用？组合后是否合理？ | 1-5 |
| **边界清晰度** | 每条规则的边界是否清晰？ | 1-5 |
| **实操可行性** | 用这个框架能否写出「像作者风格」的文章？ | 1-5 |

### 输出格式

```json
{
  "judge_score": {
    "internal_consistency": 4,
    "coverage_completeness": 3,
    "composability": 4,
    "boundary_clarity": 5,
    "practical_feasibility": 4
  },
  "overall_score": 4.0,
  "issues": [
    {
      "dimension": "coverage_completeness",
      "problem": "缺少中段（第15-25章）的决策规则",
      "suggestion": "补充中段的情绪控制规则"
    }
  ],
  "verdict": "PASS_WITH_ISSUES"
}
```

### 判定标准

| 总分 | 判定 | 处理 |
|------|------|------|
| ≥4.5 | PASS | 直接交付 |
| 4.0-4.4 | PASS_WITH_ISSUES | 标注问题，可选修复 |
| 3.5-3.9 | NEEDS_REVISION | 回炉 Phase 4 |
| <3.5 | REJECT | 回炉 Phase 2 |
```

### 3.3 系统化迭代（P1）

**目的**：让蒸馏工具链持续进化

**现状**：Phase 8 已有「后蒸馏回馈」，但只做缺口检测和自动修正

**增强方案**：增加「蒸馏效果追踪」和「工具链版本管理」

```markdown
## Phase 8 增强：系统化迭代

### 8d：蒸馏效果追踪

每次蒸馏后，记录本次蒸馏的「产出质量指标」：

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| **规则存活率** | Phase 3 通过数 / Phase 2 提取数 | 衡量提取器精准度 |
| **质量评分均值** | Phase 3.5 所有规则的平均分 | 衡量规则质量 |
| **框架评分** | Phase 7.3 judge 总分 | 衡量整体框架质量 |
| **压力测试通过率** | Phase 6 通过数 / 总测试数 | 衡量框架可用性 |

**输出**：写入 `evolution-log.md` 的「效果指标」section

### 8e：工具链版本管理

每次蒸馏后，如果修改了 extractor prompt 或阈值，记录版本变化：

```markdown
## 工具链版本日志

| 日期 | 修改对象 | 修改内容 | 触发原因 | 效果验证 |
|------|---------|---------|---------|---------|
| 2025-06-02 | mental-model-extractor.md | 增加「信念形成路径」维度 | Agent B 反馈 | 待下次蒸馏验证 |
| 2025-06-02 | decision-heuristic-extractor.md | 强化案例深度要求 | Agent B 反馈 | 待下次蒸馏验证 |
```

**目的**：
1. 追踪哪些修改有效、哪些无效
2. 支持回滚（如果修改产生副作用）
3. 形成「修改→验证→调整」的闭环
```

## 六、总结

### 改造原则

1. **精简优先**：不新增独立提取器，而是增强现有提取器
2. **分级实施**：P1 必做（双Agent、Judge、质量评分、迭代），P2 可选（提取器增强、7级扫描）
3. **闭环设计**：每次蒸馏都产出「效果指标」，支持持续优化

### 预期效果

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 规则质量 | 有验证，无量化评分 | 有量化评分，可筛选 |
| 框架评估 | 无 | 有框架级 Judge |
| 迭代能力 | 有缺口检测 | 有效果追踪+版本管理 |
| 提取器覆盖 | 8个提取器 | 8个提取器 + 可选增强 |
