# story-rewrite：结构仿写

**核心信念：只仿"骨相"，不仿"皮相"。**

**架构原则：主线程预读一切，agent 拿到 prompt 就写，不读任何文件。**

---

## ⚠️ 核心约束

### 反调色盘（防止"扒骨肉"）

仿写必须在以下3个维度上与源文本**完全不同**：

| 维度 | 说明 |
|------|------|
| **开篇方式** | 如源文本用心理治疗/催眠开篇，新书必须用其他方式（商战/遗物/急救等） |
| **破冰方式** | 如源文本用送物被拒，新书必须用其他方式（替解围/共同任务/意外接触等） |
| **节奏安排** | 如源文本一章写重生前，新书必须压缩或改变节奏 |

### 时间线连续

Phase 1 框架只提供「情绪弧线+叙事功能」，不提供具体场景。Phase 2 生成章纲锁定关键事件。Phase 3 agent 在章纲内自由发挥。

---

## 流程总览

```
Phase 1（主线程，~2min）  Phase 2（1 agent，~2min）  Phase 3（3 agent 并行，~3min）  Phase 4（主线程，~30s）
读源文本 + 提取框架        生成章纲（每章情节点）        仿写3章（prompt即写，不读文件）   合并TXT + 字数统计 + 自检
```

**总耗时：~7-8分钟（试水3章）**

### 长篇扩展模式（>3章）

```
批次1：Phase 1-4 写第1-3章 → 提取摘要
批次2：Phase 2-4 写第4-6章 → prompt注入前3章摘要
批次3：Phase 2-4 写第7-9章 → prompt注入前6章摘要
...
```

每批之间主线程做两件事：
1. 压缩前文摘要（角色状态、未解伏笔、当前时间点）
2. 规划下一批的章纲

---

## Phase 1：结构提取 + 预加载

**用户只提供源文本路径，剩下全部自主完成。**

主线程收到源文本后，**同步完成以下操作**（不启动 agent，直接在主线程做）：

### Step 1：读取源文本

读取前3章 + 最后1章，记录：
- 源书名、题材、章数
- 开篇方式（具体场景 → 抽象为叙事功能）
- 破冰方式（具体事件 → 抽象为叙事功能）
- 节奏安排（各章情绪基调）
- 人设标签（女主/男主/配角/反派）
- 结局走向

### Step 2：预加载参考文件

同步读取以下两个文件，提取关键内容嵌入 Phase 2 prompt（agent 不再读文件）：

1. `narrative-writer.md`（位于 `.opencode/agents/`）→ 提取**三层展开法**（发生层+感知层+反应层）和**情绪弧线规则**
2. `banned-words.md`（位于 `skills/story-deslop/references/`）→ 提取**一级禁用词**（情态/动作/表情/心理/判断/形容/过渡类，**控制在 20 词以内**，避免挤占 Agent 推理空间）

⚠️ 主线程读取时需拼接项目绝对路径，不能用 `~/.opencode/`（那是用户全局目录，文件不存在）。

### Step 3：生成仿写框架

输出以下框架（只保留抽象模式，禁止具体场景）：

```
- 新书名：
- 新主角：（名字+核心标签，必须与源文本不同）
- 新背景：（时代/世界观，必须与源文本不同时代/不同地域）
- 核心冲突：（1句话）
- 开篇叙事功能：（抽象，如「铺垫→创伤回忆→重生契机」）
- 破冰叙事功能：（抽象，如「主动靠近→遭遇冷遇→坚持」）
- 开篇差异点：（必须与源文本不同）
- 破冰差异点：（必须与源文本不同）
- 节奏对齐表（3章）：
  - 第1章：情绪基调 / 叙事功能 / 弧线走向
  - 第2章：情绪基调 / 叙事功能 / 弧线走向
  - 第3章：情绪基调 / 叙事功能 / 弧线走向
- 人物关系模式（抽象）：
- 信息差设计：
- 核心差异对照表（时代/地域/职业/身份层面）：
- 角色名表（主线程必须统一，禁止 Agent 自行改名）：
  - 女主：{名字}
  - 男主：{名字}
  - 配角A：{名字+功能}
  - 配角B：{名字+功能}
- 题材标签：

## 关键事件锚点表（Phase 1 主线程填写，Phase 2 章纲展开，Phase 3 agent执行）
- 第1章锚点：{必须发生的事件A}，{必须发生的事件B}，本章以{悬念/情绪点}收尾
- 第2章锚点：时间线{紧接第1章/次日/数日后}，{必须发生的事件A}，{必须发生的事件B}
- 第3章锚点：时间线{紧接第2章/次日/数日后}，{必须发生的事件A}，{必须发生的事件B}
```

**自检**：框架给没读过源文本的人，能据此写出完全不同但结构相似的故事吗？不能 → 重新抽象化。

---

## Phase 2：章纲生成

框架生成后，启动 1 个 `task(subagent_type: "general")` 生成章纲。

**章纲的作用**：锁定每章的关键事件，防止多agent并行时各自为政导致故事断裂。

### 章纲生成 prompt


### 主线程提取章纲

agent返回后，主线程提取每章的：
- 章名
- 时间线
- 核心事件
- 情节点序列
- 目标情绪
- 章首钩子
- 爽点
- 章尾钩子（含期待度）
- 字数目标

这些内容将填入Phase 3的agent prompt中。

**关键优化：主线程预读一切，agent 不读任何文件。**

每个 agent 的 prompt 由以下模板生成（主线程填入变量后发送）：

**⚠️ 字数硬约束：每章必须控制在 1600-2400 字（基准 1800-2200，浮动 10%）。不达标则全文重写，最多重写 3 次。**

```
- 目标：2000 字（范围 1600-2400，基准 1800-2200 浮动 10%）

4. 只返回状态信息，不要返回正文：
```

### ⚠️ 关键事件锚定规则（防止多agent各自为政）

**问题根因**：多agent并行时，如果prompt没有锁定关键事件，每个agent会自行设计，导致章节间时间线断裂、事件版本冲突、人物行为矛盾。

**解法**：
1. Phase 1 框架包含「关键事件锚点表」，主线程在生成框架时把每章必须发生的1-2个关键事件锁死
2. Phase 2 生成章纲，将锚点展开为具体的情节点（时间线+事件+钩子）
3. Phase 3 agent prompt 注入章纲内容，agent在章纲内自由发挥

锚点是「叙事功能+关键事件」层面的抽象描述，章纲是具体的情节点，agent在章纲框架内自行设计场景。

### 主线程构建 prompt 时的变量填充

| 变量 | 说明 |
|------|------|
| `{N}` | 章节号（1/2/3） |
| `{前情摘要}` | 第1章留空；第2章写「第1章结尾：{一句话}」；第3章写「前两章进展：{两句话}」 |
| `{框架内容}` | Phase 1 输出的完整框架（含角色名表+关键事件锚点表） |
| `{章纲内容}` | Phase 2 输出的当前章节章纲（章名+核心事件+情节点序列+目标情绪+章首钩子+爽点+章尾钩子+字数目标） |
| `{禁用词列表}` | 从 `banned-words.md` 提取的一级禁用词（情态/动作/表情/心理类，控制在 20 词以内，避免挤占推理空间） |

Phase 3 的 agent 返回后，主线程从 `temp_{书名}_ch{N}.txt` 读取各章正文，**一口气完成**：

## 踩坑记录

### Windows 中文编码问题（2025-05-30）

**现象**：`python -c` 执行包含中文的三引号字符串时，PowerShell 编码与 Python 不一致，导致 `SyntaxError: unterminated triple-quoted string literal`。

**根因**：Windows PowerShell 5.1 默认编码（GBK/cp936）与 Python 的 UTF-8 不匹配，中文字符被截断。

**解法**：
- 文件写入 → 用 `write` 工具，不用 `bash` + `python -c`
- 字数统计 → 先 write 文件，再单独 `python wordcount.py` 读文件统计
- 绝不在 `python -c` 命令里包含多行中文字符串

### 多agent并行导致故事断裂（2025-05-30）

**现象**：多个agent并行写多章，每个agent自行设计关键事件，导致章节间时间线断裂、事件版本冲突、人物行为矛盾。

**根因**：Phase 2 prompt 没有锁定关键事件锚点，agent各自发挥。

**解法**：
- Phase 1 框架新增「关键事件锚点表」，主线程锁定每章必须发生的1-2个关键事件
- Phase 2 新增章纲生成步骤，将锚点展开为具体情节点（对齐 story-long-write 细纲格式：情节点序列+钩子+爽点+情绪目标）
- Phase 3 prompt 注入章纲内容，agent在章纲内自由发挥
- 锚点是抽象的「叙事功能+关键事件」，章纲是具体情节点，agent写具体场景

### story-rewrite 章纲与 story-long-write 细纲对齐（2025-05-30）

**问题**：story-rewrite 章纲只有2个关键事件+收尾钩子，粒度太粗，缺少钩子设计、爽点、情绪目标。

**对齐方案**：章纲格式对齐 story-long-write 细纲，包含：
- 核心事件（1句话）
- 情节点序列（3-5个，每个写清"谁做了什么"）
- 目标情绪
- 章首钩子
- 爽点
- 章尾钩子（含期待度）
- 字数目标

### opencode 逐字输出导致长文本慢（2025-05-30）

**现象**：agent 返回长正文时，opencode 逐字输出到终端，几百字就要等很久。

**解法**：
- agent 用 write 工具写入 `temp_{书名}_ch{N}.txt`，不返回正文到终端
- agent 只返回状态信息（章名+字数+达标状态）
- 主线程从文件读取正文合并

### 多任务并行导致临时文件覆盖（2025-05-30）

**现象**：同时开多个 rewrite 任务时，`temp_ch1.txt` 互相覆盖，导致合并时内容错乱。

**解法**：
- 临时文件用新书名作前缀：`temp_{书名}_ch{N}.txt`
- 书名从框架中取，去掉特殊字符
- 合并完成后删除临时文件
