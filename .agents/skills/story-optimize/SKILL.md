# story-optimize

> 审稿反馈 → 深度追溯根因 → 精准修正（而非堆规则）

## 精读方法论

读报告不是摘抄问题，是**诊断**。每一条反馈追问三层：

### 第一层：现象层
报告在说什么？
- 抄原文症状：报告说"雷同""相似""一致"
- 质量症状：报告说"节奏慢""人设平""AI味"
- 这是症状，不是病因

### 第二层：溯源层
这个问题的直接来源是什么？

| 根因类别 | 定义 | 典型表现 | 解法方向 |
|---------|------|---------|---------|
| **数据泄露** | LLM 看到了不该看的东西 | 源文细节/编号/设定词出现在新书 | 改进 source_stripper 脱敏强度 |
| **方法矛盾** | 设计目标的内在冲突 | "仿骨架"和"纯原创"无法兼得 | 明确仿写边界，区分"骨架"和"血肉"的定义 |
| **指令模糊/错位** | prompt 没说清楚优先级 | LLM 知道规则但选了错误的平衡 | 精简指令，去掉冗余，标清优先级 |
| **流程缺失** | 某个必要步骤不存在 | 跨章衔接差、风格不一致 | 加衔接修复阶段或后处理 |
| **测量偏差** | 审稿工具/人的判断不准 | 误报抄袭、标准不一致 | 了解审稿的上下文，判断是真问题还是误报 |
| **模型噪声** | LLM 随机翻车 | 单个章节异常，周边章节正常 | 重跑，不沉淀规则 |
| **副作用耦合** | A修复导致B损坏 | 改prompt后某类问题减少但新问题出现 | 拆解修复，隔离影响范围 |

### 第三层：架构层
系统性还是偶发性？

| 分布特征 | 诊断 | 行动 |
|---------|------|------|
| 单章偶发 | 模型噪声 | 重跑，不过度设计 |
| 同模式跨2-3章 | 局部指令模糊 | 调整对应 prompt 的优先级 |
| 同模式跨≥4章 | 流程级/数据源问题 | 改 pipeline 或脱敏层 |
| 全10章均匀分布 | 设计目标本身的问题 | 需要重新定义"仿写"边界 |
| 随版本迭代恶化 | prompt 退化（规则太多压垮核心指令） | 大精简，砍掉冗余规则 |

### 多报告交叉分析

当有≥2份报告时，**必须**做交叉对比：

```
报告A说____ 报告B说____
  ├─ 一致且合理 → 确认是真问题
  ├─ 一致但可能误报 → 两个审稿者犯了同样的偏见
  ├─ 分歧 → 谁更可信？
  │    ├─ 报告的说理更细致 → 倾向它
  │    ├─ 报告来自更权威的来源 → 倾向它
  │    └─ 无法判断 → 看原文实际内容
  └─ 分歧原因分析
       ├─ 审稿标准不同（宽松 vs 严格）
       ├─ 审稿人/工具不知道"章对章仿骨架"是设计目标
       └─ 关注点不同（细节 vs 结构）
```

**判断准则**：如果两份报告对**事实**一致（都认为结构对应），但对**结论**分歧（一中高一低），根本原因通常是审稿者不知道项目的设计目标。这不是你 prompt 的问题，是评估框架的问题。

## 流程（防守 + 进攻）

### 防守线（问题修复）
```
收集报告 → 报告分类 → 交叉对比 → 分层诊断 → 定位根因层级 → 选择解法 → 沉淀/pruning → 验证
```

### 进攻线（品质提升）
报告的"低分项"不一定是有问题——更可能是"不够好"。

```
读报告 → 找"中/低评价"项 → 判断：是真问题，还是"不够好"？
  ├─ 真问题 → 走防守线
  └─ 不够好 → 问：让这个维度从6分到9分，prompt需要什么？
       ├─ 更强调某个方向？（如"记忆点"）
       ├─ 需要系统性的写法指导？（如"怎么制造张力"）
       └─ 还是纯模型能力差异？（如文笔细腻度→不处理）
```

**防守线修"不犯错"，进攻线修"不够好"。两个都要过。**

**读取来源**：用户粘贴内容 / `compare/对比_*_AI分析.md` / `compare/对比_*_报告.md`

## 报告分类

不同来源的报告可信度和侧重点不同：

| 报告类型 | 可信度 | 擅长发现 | 不擅长 | 使用时注意 |
|---------|-------|---------|-------|-----------|
| **用户直接反馈** | 最高 | 真实阅读体验的问题 | 可能主观 | 作为首要依据 |
| **AI分析版报告** | 中高 | 细节对比、语言模式分析 | 不理解设计意图，容易把"仿骨架"当抄袭 | 需要过滤"方法矛盾"类误报 |
| **报告版报告** | 中 | 字数/占比等量化指标 | 定性判断弱 | 只看数据不看结论 |
| **审稿review** | 中高 | 读者视角的节奏/人设/钩子 | 可能遗漏技术细节 | 补充叙事质量评估 |

**重要**：AI分析版报告不区分"框架参考"（设计允许）和"实质抄袭"（设计禁止），它的"高风险"标签不一定代表真正的问题。

## 解法层级（选最轻的）

| 层级 | 方法 | 适用场景 | 副作用 |
|------|------|---------|--------|
| **级别1: 重跑** | 不改任何东西，再跑一次 | 单章偶发翻车 | 零 |
| **级别2: 改脱敏** | 增强 source_stripper | 数据泄露类问题 | 轻度 |
| **级别3: 改prompt** | 精简/重写 prompt 指令 | 指令模糊/优先级错乱 | 中 |
| **级别4: 改流程** | 增删 pipeline 阶段 | 流程缺失/顺序错误 | 高 |
| **级别5: 改设计目标** | 重新定义"仿写""换皮"边界 | 方法矛盾类 | 极高 |

**规则**：从级别1出发，能不上就不上。必须说明：为什么不选更轻的？

## 精读范式

强制按此格式逐条分析：

```
### 问题：【问题名】

现象：报告说____
溯源：（数据泄露/流程缺失/指令模糊/模型噪声/方法矛盾/测量偏差/副作用耦合）
架构：（单章偶发/跨N章局部/系统性/全10章均匀分布/随版本恶化）
解法层级：级别N — ____
为什么不选更轻的：____
沉淀方式：（规则/脱敏/流程/重跑/_不处理_）
```
精读决策树：
```
报告说"雷同/相似"
├─ 是具体数字/编号/人名一致吗？
│   └─ 是 → 级别2: 改脱敏，在 stripper 里替换
├─ 是情节结构对应吗？
│   ├─ 这是"骨架仿写"的预期产出吗？
│   │   ├─ 是 → 不处理，属于设计允许的框架参考
│   │   └─ 否 → 才考虑修改
│   └─ 需要判断"骨架"和"血肉"的边界
├─ 是特定装置/动作链一致吗？
│   └─ 确认是"血肉"层面的一致 → 级别3: 改 plot-guide 的禁止清单
└─ 是报告误报（如 AI 分析版不理解仿写意图）？
    └─ → 不处理，标注"测量偏差"
```

## 跨章节模式识别

每次沉淀前，检查报告中有无跨章模式：

```
跨章模式_模板：

模式描述：____（如"第3/6/8章都在场景开头使用了____"）
涉及章节数：____
根因推测：____（数据泄露/prompt退化/设计方法矛盾/其他）
解法层级：____
```

**经验模式库**（持续积累）：

| 模式 | 典型表现 | 常见根因 | 推荐解法 |
|------|---------|---------|---------|
| **编号泄露** | 源文数字出现在新书 | 数据泄露 | 级别2: 改 stripper |
| **设定词泄漏** | "监狱"出现在精神科故事里 | 数据泄露 | 级别2: 改 stripper 加场景词替换 |
| **连续低信息密度** | 连续2-3章无新信息释放 | 指令模糊 | 级别3: 强化 plot-guide 的信息释放要求 |
| **角色台词模式** | 多个角色说同样句式的话 | prompt退化/模型特性 | 级别3: 精简 prompt，去掉冗余约束 |
| **跨章权重漂移** | 某条规则在前3章被执行，后面被忽略 | prompt退化 | 级别3: 检查规则是否太多，精简 |
| **同一骨架桥段被重复标记** | 审稿反复说某章的结构相似 | 测量偏差 | 不处理，标注为"设计意图" |

## Pruning 检查清单

**每次沉淀后，必须做**：
1. prompt 文件总行数？>300行则必须精简
2. 是否有≥3条规则在讲同一件事？→ 合并
3. 是否有规则可以被脱敏层替代？→ 删规则，改 stripper
4. 是否有规则来自单章偶发？→ 删，重跑解决
5. 本次规则下次触发频率预测？从未发生的→删
6. 每条规则后标注"上次触发时间"（如：规则X上次在2024-06-11触发）

---

## 输出

1. 明确标注：**根因** | **解法层级** | **沉淀位置** | **是否设计意图**
2. 报告交叉分析结论（如果有多个报告）
3. 记录删除/合并了几条旧规则
4. 向用户报告：现象→溯源→是否设计意图→解法→预期效果
