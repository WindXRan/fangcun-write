# 逆推法 — 回归测试管线

**核心方法论：** 用优质原书章节作为测试用例，验证"原文→三纲→正文"整个管线的读者体验是否无损。
**如果逆推正文在读者体验上和原文相当 → prompt 已学到这本书的写作配方 → 用于写原创章节也是同级别的。**

---

## 什么是逆推法

```
            逆  推  法
    ┌──────────────────────────────────┐
    │                                  │
    │   原文  ─→  三纲  ─→  逆推正文   │
    │   (已知好)  (结构化)  (独prompt)  │
    │                                  │
    │          ↕ 对比读者体验 ↕          │
    │   差距大 → 说明 prompt 没学会     │
    │   差距小 → prompt 可量产爆款      │
    │                                  │
    └──────────────────────────────────┘
```

**三纲 = 总纲 + 卷纲 + 章纲 + 角色 + 设定（结构化数据的统称）**

逆推法回答一个问题：**"如果我只给 AI 三纲，不给原文，AI 能不能写出和原文一样好看的章节？"**

能 → prompt 学会了。不能 → 缺的东西就是优化方向。

## 为什么有用

传统 prompt 优化是"写一版→凭感觉改→再写一版"，永远不知道改对了没有。

逆推法把过程变成可测量的：

| 传统方式 | 逆推法 |
|---------|--------|
| "这章感觉不太行" | 具体哪个维度弱？代入感？主角辨识度？ |
| "改一下看看" | 一次只修 1-2 个维度，修完跑对比验证 |
| "好像好点了" | 看对比表的 A>B/A≈B/A<B 数量变化 |
| "不知道哪里还有问题" | 对比表里原文还赢着几个维度，就是下一步目标 |

## 迭代循环：分析 → 补丁 → 验证 → 追踪

**核心原则：检测不是为了看分数，是为了出补丁。**

```
                    ┌─────────── 迭代循环 ───────────┐
                    │                                  │
     Round 前 ──→ Step 1~2: 跑对比 ──→ Step 3: 聚合分析 ──→ Step 4: 出补丁
        ↑                                              │
        │                                              ▼
        └── Round 后: 更新注册表 ←── Step 6: 下一轮验证 ←── Step 5: 应用补丁
```

关键规则：
- **每一轮迭代的实际产出 = 1-2 个补丁写入 prompt 文件 + 验证结果**
- 如果一轮结束后没有修改任何 prompt 文件 → 这轮没有产生价值
- 补丁必须精确到 old_string / new_string，能直接用 Edit 工具执行

---

### Round 前：准备

```
1. 读 _optimize/anomaly_patterns.md → 查看哪些模式是 🔴 active
2. 读 _optimize/fix_registry.md → 查看上次补丁的验证状态
3. 如果上轮有 ⏳ 待验证 的补丁 → 本轮先用 compare 验证，不出新补丁
4. 从 active 模式中选 1 个复现最久的 → 作为本轮修复目标
```

### Round 中：四步循环

#### Step 0: 确定本次测试的章节和基线
```
指定：第N章 + 固定原始输出（或原文）
```

#### Step 1: P0 规则检查（0 API 调用）
```
python checker.py --output 正文/正文/第N章.xml

结果：
├─ 全部 pass → 进 Step 2
└─ 有 fail  → 先修输出再跑（不修到 pass 不值得对比）
```

#### Step 2: 维度对比（1 次 LLM 调用）
```
python compare.py \
  --original 源文/第N章.txt \
  --output 正文/正文/第N章.xml \
  --guide 正文/章纲/第N章.xml

输出：
├─ 结构对齐表（5维: 一致/不一致）
└─ 读者体验对比表（7维: A>B/A≈B/A<B）
```

#### Step 3: 聚合分析 → 产补丁计划
```
当对比结果显示原文仍领先时，而非直接人工判断：

1. 收集数据源：
   ├─ 本轮 compare 输出
   ├─ 近 3-5 章 fanxie-review 报告（如有）
   ├─ _optimize/anomaly_patterns.md（active 模式）
   └─ _optimize/fix_registry.md（历史已应用补丁）

2. 运行 fanxie-prompt-opt（1 次 LLM 调用）：
   → 输出 _optimize/prompt_regression/round{N}/prompt_patch_ready.md
   → 包含：old_string + new_string + 目标文件 + 优先级
   → 每条补丁是可直接执行的文件修改操作
```

#### Step 4: 应用补丁（这才是本轮核心产出）

```
逐条执行补丁（使用 Edit 工具）：

1. 读目标文件当前状态（必须读，不是假设）
2. 定位 old_string → 确认存在于文件中
3. 用 Edit 工具替换为 new_string
4. 如 old_string 不在文件中 → 说明文件已被其他轮次修改 → 找实际存在的锚点文本
5. 每应用一条，更新 _optimize/fix_registry.md：
   - 补丁 ID: CH-NNN
   - 应用轮次: R{N}
   - 目标文件: 路径
   - 实际替换内容: (old → new)
   - 状态: ✅ solved（已应用，待验证）

限制：
├─ 每轮最多改 1-2 个 P0 补丁
├─ 不允许同时改 3 个文件
└─ 改完必须确认文件保存成功
```

#### Step 5: 验证补丁效果

```
1. 重新跑 Step 1（checker）→ 确认格式红线未破坏
2. 重新跑 Step 2（compare）→ 对比修改前后的差距变化
3. 如果目标维度追平或接近 → 补丁有效
4. 如果目标维度无变化 → 补丁约束力不够 → 记录到 anomaly_patterns 并计划升版
```

### Round 后：更新追踪文件

```
1. 更新 _optimize/anomaly_patterns.md：
   ├─ 修改对应模式的 "修复历史" 表 → 追加本轮方案+效果
   ├─ 如果有效 → 标记 dormant（观察 2 轮后若不复现→resolved）
   ├─ 如果无效 → 标记 active + 记录升版方案
   └─ 如果新模式出现 → 用模板新建记录

2. 更新 _optimize/fix_registry.md：
   ├─ 本轮应用的补丁标记 ✅ solved
   ├─ 已验证效果的补丁标记验证评分
   └─ 确认上轮补丁是否有效

3. 清理：
   ├─ 本轮产物存到 _optimize/prompt_regression/round{N}/
   └─ 报告保留最近 5 轮
```

---

### 全流程对照表

| 步骤 | 做什么 | 工具/文件 | 产出 |
|------|--------|----------|------|
| Round前 | 读异常模式 + 补丁注册表 | `anomaly_patterns.md`, `fix_registry.md` | 本轮修复目标 |
| Step 0 | 定基线 | — | 测试章节号 |
| Step 1 | 格式红线检查 | `checker.py` | check_report |
| Step 2 | 维度对比 | `compare.py` | compare.json |
| Step 3 | 聚合分析出补丁 | `fanxie-prompt-opt` | prompt_patch_ready.md |
| **Step 4** | **应用补丁到 prompt 文件** | **Edit 工具** | **修改 prompt 文件** ← 核心产出 |
| Step 5 | 验证补丁效果 | checker + compare | 验证报告 |
| Round后 | 更新追踪文件 | anomaly_patterns + fix_registry | 更新后的注册表 |

**再次强调**：如果 Step 4 没有执行（没有修改任何 prompt 文件），这一轮就等于白跑了。

## 工具使用

### checker.py（P0 规则检查，0 API 调用）

```bash
python _optimize/prompt_regression/checker.py \
  --output "正文/正文/第1章.xml" \
  --report "_optimize/prompt_regression/round1/check_report.json"
```

当前检查项：

| 检查 | 方法 | 阈值 |
|------|------|------|
| 章尾总结升华 | 正则匹配最后100字 | 0 处 = pass |
| 章首钩子 | 正则匹配首50字开场方式 | 非天气/风景/日常 = pass |
| 禁用词 | 正则匹配"仿佛/眼中闪过/微微"等 | ≤3 处 = pass |
| 情绪直述 | 正则匹配"他感到/他意识到" | ≤2 处 = pass |
| 元信息词 | 正则匹配"上一章/本章/伏笔" | 0 处 = pass |
| 格式规则 | 破折号/省略号/他说她道/感叹号密度 | ≤2 项异常 = pass |
| 弱化副词密度 | 微微/淡淡/缓缓/轻轻 每千字计数 | ≤3 次/千字 |
| 段落长度 | 段均句数 + 超长段落数 | 无超过5句段落 = pass |
| 万能比喻 | 像…般/如…般/潮水般 | 0 处 = pass |
| 字数 | 去空白字数统计 | info，不判断 |

### compare.py（维度对比，1 次 LLM 调用）

```bash
python _optimize/prompt_regression/compare.py \
  --original "projects/作者/书名/正文/正文/第1章.txt" \
  --output "正文/正文/第1章.xml" \
  --guide "正文/章纲/第1章.xml" \
  --report "_optimize/prompt_regression/round1/compare.json"
```

两段式输出：

**结构对齐（5维）：** 原文 vs 输出在技术指标上是否一致

| 维度 | 取值 |
|------|------|
| 开篇手法 | action/dialogue/suspense/emotion/exposition |
| 章尾钩子 | crisis/sudden_reveal/cliffhanger/summary/emotional/countdown |
| 情绪弧线 | 起点→终点 |
| 信息释放节奏 | 前25%释放了什么 |
| 调性 | 口语/吐槽/严肃/文艺 |

**读者体验对比（7维）：** 从读者视角，原文(A)和输出(B)哪个更好

| 维度 | 含义 |
|------|------|
| 代入感强度 | 前300字是否快速进入故事、在意角色 |
| 主角辨识度 | 遮住角色名能认出是谁在说话吗？ |
| 对话自然度 | 对话像真人说话还是功能工具？ |
| 爽感密度 | 读完觉得爽了吗？ |
| 钩子力度 | 结尾想看下一章吗？ |
| 信息紧凑度 | 每句话都在推进吗？有注水吗？ |
| 情感共鸣 | 有情绪波动吗？（生气/开心/紧张/感动） |

## 何时跑

| 场景 | 跑什么 | 预期投入 |
|------|--------|---------|
| 改了某个预设的 prompt | 选 1 章跑 checker + compare | 1 次 LLM 调用 |
| 改了章纲模板结构 | 跑 3 章（首/中/尾各 1）| 3 次 |
| 改了角色卡格式 | 跑含该角色的章节 | 1-2 次 |
| 批量改了一组预设 | 跑 5 章代表性章节 | 5 次 |

建议：每次跑完后把对比报告存入 `_optimize/prompt_regression/round{N}/`，方便回头看 C 是否误判。

## 文件结构

```
_optimize/
├── anomaly_patterns.md          ← 📌 异常模式库（跨轮次积累，追踪复现/休眠/解决）
├── fix_registry.md              ← 📌 补丁注册表（追踪每个补丁从提出到验证的全周期）
└── prompt_regression/
    ├── checker.py                ← P0 规则检查器（纯正则）
    ├── compare.py                ← 维度对比器（1 次 LLM 调用）
    ├── round{N}/
    │   ├── output_ch{N}.xml     ← 输出正文
    │   ├── check_report.json    ← checker 输出
    │   ├── compare.json         ← compare 输出
    │   ├── prompt_patch_ready.md ← 📌 fanxie-prompt-opt 产出的可执行补丁（含 old/new_string）
    │   ├── fix_applied.md       ← 本轮修复记录
    │   └── review.md            ← 审稿报告（可选）
    └── README.md

tools/builtin/
├── 仿写/
│   ├── fanxie-chapter.md        ← 📌 仿写写章主 prompt（补丁的主要目标文件）
│   ├── fanxie-prompt-opt.md     ← 📌 聚合分析→出补丁（迭代循环的中心）
│   ├── fanxie-blindspot.md      ← 📌 盲区检测（fanxie-prompt-opt 的输入源）
│   │   ├── fanxie-review.md         ← 读者视角审查
│   └── fanxie-fix.md            ← 靶向修复
└── 写作/
    ├── write-chapter.md          ← 写章助手（含反推自检）
    ├── source-guide-reverse.md   ← 章纲逆推
    └── source-guide-reverse-validate.md ← 📌 章纲逆推质量验证
```
