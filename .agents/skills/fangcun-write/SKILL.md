# 方寸工作流合约

**以下规则为硬约束，禁止违反。**

---

## 写第N章：三纲法，缺一不可

```
Step 0: 确定卷位置
  → 查看 正文/卷纲/ 确定第N章属于第几卷
  → 读对应卷的卷纲，了解本章在卷弧线中的位置
  → 如果卷纲不存在 → 先建卷纲，再写章

Step 1: preflight
  → quality_checker.py check --chapter N
  → exit code != 0 时禁止继续

Step 2: 源文章纲提取（不允许手动编写章纲）
  → source-guide-reverse 从源文正文提取结构
  → 输出到源文项目 正文/章纲/第N章.xml

Step 3: 章纲转换（不允许直接使用源文章纲）
  → guide-convert 将源文章纲转换为仿写章纲
  → 输出到仿写项目 正文/章纲/第N章.xml
  → 检查 tool 属性必须为 "guide-convert"

Step 4: 写正文（不允许手动编写正文）
  → write-chapter 按仿写章纲写正文
  → 输出到仿写项目 正文/正文/第N章.xml

Step 5: 审查修复
  → fanxie-review（读者视角审查）
  → fanxie-structural-fix（如结构偏离章纲）
  → fanxie-fix（如技法偏差）

Step 6: 质量门禁
  → quality_checker.py audit project_dir
  → exit code != 0 时阻塞，必须修复到通过

Step 7: 展示前500字+尾200字给用户确认
  → 用户说"可以"才进入下一章
```

## 禁止事项（违反视同违约）

```
❌ 禁止手动编写章纲（必须用 source-guide-reverse）
❌ 禁止直接使用源文章纲（必须用 guide-convert 转换）
❌ 禁止手动编写正文（必须用 write-chapter）
❌ 禁止跳过 quality_checker
❌ 禁止跳过用户确认直接下一章
❌ 禁止在仿写中添加源文没有的背景/情节/人物内心
❌ 禁止缩写或扩写源文的信息释放节奏
🔴 禁止洗稿：同一功能不能用和源文相同的句式/措辞/修辞
   ➜ 源文写"主角投胎了。她刚刚被分娩出来，就奋力睁大了眼睛"
   ➜ 洗稿："仿写主角投胎了。她刚刚被分娩出来，就奋力睁大了眼睛" ❌
   ➜ 仿写："仿写主角努力撑开黏糊糊的眼皮，入目是一室烛光" ✅
   ➜ 功能相同（分娩睁眼+环境），措辞完全不同
```

## 允许事项

```
✅ 角色名/称谓/势力名/地名/时代背景的换皮
✅ 保留源文的情绪弧线和信息释放点（必须一致）
✅ 保留源文的节奏密度（段落数、OS密度、对话比例）
✅ 用不同的语言表达相同的情节功能（这才是仿写）
```

## 质量门禁

```bash
# 写前检查（阻塞）
python quality_checker.py check project_dir --chapter N

# 写后审计（阻塞）
python quality_checker.py audit project_dir
```

`exit code 0` = 通过。`exit code 1` = 阻塞，必须先修复。

## 🧠 Agent 智能行为规则（必须遵循）

### 1. 【项目状态扫描】每次任务开始前必须执行

**强制规则**：收到任何写作任务时，第一步必须是扫描项目状态并生成报告。

#### 📋 扫描步骤（必须按顺序执行）

```
Step 1: 扫描已写章节
→ 使用 Glob 工具：glob(pattern="正文/正文/第*.xml")
→ 提取章节号，排序，得到 written_chapters = [1, 3, 4, 5, ...]
→ 如果正文/正文/ 目录不存在 → 返回空列表

Step 2: 扫描章纲
→ 使用 Glob 工具：glob(pattern="正文/章纲/第*.xml")
→ 提取章节号，排序，得到 guides = [1, 2, 3, ...]
→ 如果正文/章纲/ 目录不存在 → 返回空列表

Step 3: 识别章节间隙
→ 遍历 written_chapters，找出所有"不连续"的位置
→ 例如：written_chapters = [1,2,3,6,7,8] → gaps = [(3,6)] 表示第4-5章空缺
→ 如果没有间隙 → gaps = []

Step 4: 生成项目状态报告
→ 必须输出以下格式的报告（不能省略）
```

#### 📊 项目状态报告格式（必须输出）

```
【项目状态报告】

✅ 已写章节：X章
   - 章节列表：[1, 2, 3, ..., N]
   - 最新章节：第N章

✅ 已生成章纲：X章
   - 章节列表：[1, 2, 3, ..., M]
   - 覆盖情况：{有章纲但未写正文：...} {写了正文但无章纲：...}

⚠️ 章节间隙：X处
   - 间隙1：第A章 ~ 第B章之间空缺（共C章）
   - 间隙2：...
   - （如无间隙 → 显示"无章节间隙"）

📍 当前任务：写第N章
   - 位置判断：{正常顺序写作 | 填充间隙 | 跳跃式写作}
   - 建议：{根据位置判断给出具体建议}
```

#### 🔍 位置判断逻辑

```
IF N == max(written_chapters) + 1:
   → 位置 = "正常顺序写作"
   → 建议 = "直接写下一章，关联章节号自动设为上一章"

ELIF N 在某个间隙内（如 gaps = [(3,6)] 且 N = 4 或 5）:
   → 位置 = "填充间隙"
   → 建议 = "必须同时读取前置章节（第3章）章尾和后置章节（第6章）章首，确保衔接"

ELSE:
   → 位置 = "跳跃式写作"
   → 建议 = "警告：第X章之后直接写第N章，中间章节（X+1 ~ N-1）未写。建议先填充中间章节。"
```

#### ⚠️ 必须阻塞的情况

```
遇到以下情况时，必须阻塞并提醒用户：

1. 章纲缺失：要写的章节 N 没有章纲 → 必须先运行 source-guide-reverse + guide-convert
2. 跳跃过大：如已写1-3章，要写第10章 → 提醒用户确认是否跳过中间章节
3. 章节覆盖：要写的章节 N 已存在正文 → 提醒用户是覆盖还是跳过
```

### 2. 【智能上下文决策】根据章节位置自动传参

**核心逻辑**：Agent 必须根据当前章节号和项目状态，智能决策上下文参数，并在调用工具前明确输出决策原因。

#### 🎯 决策流程（必须执行）

```
Step 1: 读取项目状态报告（从 Rule 1 的输出）
   → 确定当前章节 N 的位置：{正常顺序 | 填充间隙 | 跳跃式}

Step 2: 根据位置决策上下文参数
   → 按照下方的"位置→参数映射表"决策

Step 3: 输出决策结果
   → 必须输出【上下文决策】卡片（格式见下方）

Step 4: 读取必要内容
   → 根据决策结果，调用 VariableResolver 读取 @关联章节、@后续章节 等
```

#### 📊 位置→参数映射表

| 位置类型 | 关联章节号 | 后续章节号 | 读取内容 | 特别提醒 |
|---------|-----------|-----------|---------|---------|
| **正常顺序** | N-1（上一章） | None | @关联章节、@关联章纲 | 标准衔接 |
| **填充间隙** | 间隙起始章 | 间隙结束章 | @关联章节、@后续章节、@关联章纲 | ⚠️ 必须同时衔接前后 |
| **跳跃式** | 最近的前置章节 | None | @关联章节、@关联章纲 | ⚠️ 提醒用户确认 |

#### 📋 上下文决策卡片格式（必须输出）

```
【上下文决策】

当前章节：第N章
章节位置：{正常顺序写作 | 填充间隙（第A章~第B章）| 跳跃式写作}

决策结果：
✅ 关联章节号 = X
   - 原因：{最近的前置章节是第X章 | 间隙的前置章节是第X章}
   - 将读取：@关联章节（第X章章尾，800字）

✅ 后续章节号 = Y（如有）
   - 原因：{最近的后置章节是第Y章 | 间隙的后置章节是第Y章}
   - 将读取：@后续章节（第Y章章首，500字）

✅ 关联章纲数 = 1
   - 将读取：@关联章纲（第X章章纲）

⚠️ 特别提醒：
   - {根据位置类型显示相应提醒}
```

### 5. 【质量自检】写完后必须自动检查

**强制规则**：每章写完后在展示给用户前，Agent 必须执行质量自检，并通过所有检查项才能展示。

#### 📋 质量自检清单（必须逐项检查）

```
【质量自检清单】

═══════════════════════════════════════
1. 衔接性检查（权重：高）
═══════════════════════════════════════

✅ 章首是否承接上一章结尾情绪？
   - 检查方法：读取 @关联章节 的章尾（最后300字），对比本章前300字
   - 必须通过：本章前300字必须出现"承接句"，明确延续上章情绪/悬念
   - 如不通过 → 禁止展示，先修改章首

✅ 是否读了 @关联章节 的章尾？
   - 检查方法：查看【补充信息】卡片，确认已读取关联章节章尾
   - 必须通过：必须读取，否则无法判断衔接性
   - 如不通过 → 禁止展示，先读取关联章节

✅ 前300字是否有"承接句"？
   - 检查方法：手动检查本章前300字，是否有明确承接上章的句子
   - 必须通过：例如"苏棠刚放下茶杯，便听见..."（承接上章的"茶杯"事件）
   - 如不通过 → 禁止展示，先添加承接句

═══════════════════════════════════════
2. 节奏检查（权重：高）
═══════════════════════════════════════

✅ 是否达到章纲规定的情绪弧线？
   - 检查方法：对比章纲中的 <emotional_arc> 和正文的情绪走向
   - 必须通过：正文的情绪走向必须和章纲一致
   - 如不通过 → 禁止展示，先调整情绪描写

✅ 信息释放点是否和章纲一致？
   - 检查方法：对比章纲中的 <info_release> 和正文的信息释放点
   - 必须通过：信息释放的顺序和节奏必须一致
   - 如不通过 → 禁止展示，先调整信息释放

✅ 章尾是否有悬念钩子？
   - 检查方法：读取本章最后200字，判断是否有悬念
   - 必须通过：章尾必须留悬念（钩子），让读者想看下一章
   - 如不通过 → 禁止展示，先修改章尾

═══════════════════════════════════════
3. 版权检查（权重：极高）
═══════════════════════════════════════

✅ 是否出现源文角色名？（必须用仿写名）
   - 检查方法：搜索正文中的角色名，对比源文角色名和仿写角色名
   - 必须通过：不能出现源文角色名（如"乔娇娇"），必须用仿写名（如"苏棠"）
   - 如不通过 → 禁止展示，先替换所有源文角色名

✅ 是否有和源文相同的句式/措辞？
   - 检查方法：对比源文和仿写的句式结构
   - 必须通过：不能洗稿，必须用不同的表达相同的功能
   - 如不通过 → 禁止展示，先改写相似句式

✅ 是否保留了源文节奏但换了表达？
   - 检查方法：对比源文和仿写的节奏密度（段落数、OS密度、对话比例）
   - 必须通过：节奏必须一致，但表达必须不同
   - 如不通过 → 禁止展示，先调整表达

═══════════════════════════════════════
4. 网文规律检查（权重：中）
═══════════════════════════════════════

✅ 女频：是否有情绪爆点？关系是否有变化？
   - 检查方法：读取正文，判断是否有情绪爆点（甜/虐/爽/急/酸）
   - 必须通过：每章至少有一个情绪爆点，关系必须有变化或动摇
   - 如不通过 → 禁止展示，先添加情绪爆点

✅ 男频：主角是否主动？是否有进展？
   - 检查方法：读取正文，判断主角是否主动，是否有进展（升级/打脸/布局/收获）
   - 必须通过：主角必须主动，每章至少有一个进展
   - 如不通过 → 禁止展示，先调整主角行为

✅ 通用：对话占比是否合理？OS密度是否合理？
   - 检查方法：统计正文的对话占比和OS密度
   - 必须通过：对话占比30-50%，OS密度每章2-5处
   - 如不通过 → 禁止展示，先调整对话和OS

═══════════════════════════════════════
【自检结果】
═══════════════════════════════════════

✅ 通过项：X/15
❌ 未通过项：Y/15
   - {列出所有未通过的检查项和原因}

【决策】
   - 如 未通过项 > 0 → 禁止展示给用户，先修复所有未通过项
   - 如 未通过项 = 0 → 可以展示给用户
```

#### 🔄 修复流程

```
如自检发现未通过项：

1. 输出【自检结果】，明确列出所有未通过项
2. 针对每个未通过项，提出修复方案
3. 询问用户："发现X处问题，是否自动修复？"
4. 用户确认后，执行修复
5. 修复完成后，重新执行质量自检
6. 直到所有检查项通过，才展示给用户
```

### 7. 【统筹意识】写前必执行（禁止机械编排）

**强制规则**：写第N章前，Agent 必须完成"统筹思考"，并在输出中展示【统筹思考】卡片。禁止机械地按章纲 beat 翻译。

#### 🧠 统筹思考五步法（必须执行）

```
Step 1: 位置感 — 这章在整本书的哪里？
  → 读 @总纲（前500字）→ 确定当前卷、主要矛盾
  → 读 @当前卷纲（前500字）→ 确定本卷目标、当前进度
  → 在内心回答：这章是铺垫 / 高潮 / 转折 / 收尾？
  → 在【统筹思考】卡片中输出：位置判断

Step 2: 信息流 — 这章该释放什么新信息？
  → 读 @关联章节（前3章正文最后800字）→ 提取已释放的信息
  → 在内心列出：前3章已经说了什么？
  → 确定：这章必须释放至少1条新信息（不能重复前3章）
  → 在【统筹思考】卡片中输出：新信息清单

Step 3: 情绪弧 — 这章的情绪目标是什么？
  → 读 @本章章纲 的 <emotional_arc> 标签
  → 读前3章章纲的 <emotional_arc>（对比情绪走向）
  → 在内心回答：情绪是推进了，还是原地踏步？
  → 在【统筹思考】卡片中输出：情绪目标 + 推进验证

Step 4: 角色状态 — 角色在这章该有什么变化？
  → 读 @本章章纲 的 <characters> 标签
  → 在内心列出：每个出场角色当前的状态、目标、关系
  → 确定：这章结束后，至少1个角色的状态必须有推进
  → 在【统筹思考】卡片中输出：角色推进清单

Step 5: 伏笔检查 — 前面埋了什么？这章该回收什么？
  → 快速扫前3章正文，找【伏笔】或【暗示】标记
  → 在内心列出：待回收的伏笔
  → 确定：这章是否该回收某个伏笔？是否该新埋一个？
  → 在【统筹思考】卡片中输出：伏笔回收/新埋计划
```

```

→ 正在调用 write-chapter...
```

#### ✍️ 写章时的统筹约束（必须遵循）

```
写完每一段后，在内心自检：

1. 这段释放新信息了吗？
   → 如果没有，删或改（不能浪费篇幅在重复信息上）

2. 这段推进角色状态了吗？
   → 如果没有，加戏（不能让角色原地踏步）

3. 这段的情绪是对的吗？
   → 对照【统筹思考】卡片的"情绪目标"
   → 如果偏离，调整情绪描写

4. 这段和前3章重复吗？
   → 如果重复，重写（信息不能重复释放）

5. beat 之间有统筹过渡吗？
   → 禁止机械地按 beat 翻译
   → 必须在 beat 之间加"统筹过渡"：
      - beat 1 结束前，埋下 beat 2 的引子
      - beat 2 开始后，先承接 beat 1 的情绪残渣
      - 让读者感觉"这一章是一个整体"，而不是"4个独立场景拼起来的"
```

#### ⚠️ 机械编排的检测与纠正

```
如果出现以下情况，视为"机械编排"，必须纠正：

❌ 症状1：章纲写"beat 1: 开场 - 主角起床"
         → AI 写："清晨，主角从床上醒来..."
         → 纠正：必须有统筹意识——为什么要写起床？这章的情绪目标是什么？
                  如果是"紧张"情绪，应该从"被噩梦惊醒"开始，而不是"平静起床"

❌ 症状2：章纲写"beat 2: 对话 - 主角和配角讨论"
         → AI 写：配角说"..."，主角说"...", 配角说"..."
         → 纠正：必须有统筹意识——这段对话要释放什么新信息？推进什么角色状态？

❌ 症状3：章纲写"beat 3: 转折 - 突发事件"
         → AI 写：突然，X事件发生了...
         → 纠正：必须有统筹意识——这个转折在前文有伏笔吗？没有就要先埋伏笔

❌ 症状4：章纲写"beat 4: 结尾 - 悬念"
         → AI 写：章尾留悬念...
         → 纠正：必须有统筹意识——这个悬念和本章释放的新信息有关吗？
                  不能为了悬念而悬念，悬念必须和"信息流"挂钩

✅ 正确示例（有统筹意识）：
   - 章纲：beat 1 开场
   - AI 思考：本章情绪目标是"紧张"，前3章情绪是"轻松→好奇→警惕"
             所以开场不能"平静起床"，要"被噩梦惊醒 + 联想到前章结尾的伏笔"
   - AI 写："苏棠从噩梦中惊醒，额上全是冷汗。梦里那个黑影的脸，和昨日宫门口偶遇的那人重叠在一起..."
             （承接上章伏笔 + 直接进入紧张情绪）
```
