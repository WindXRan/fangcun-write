# style-distill：文风蒸馏引擎

## 核心架构

```
原文千万字
    ↓
┌───────────────┬───────────────┐
│   知识库       │   风格理解     │
│ （可检索原文）  │ （系统提示词）  │
└───────────────┴───────────────┘
                ↓
          赛博作者（对话agent）
          - 模仿写作
          - 讨论技巧
          - 分析原文
          - 检索知识库
```

**关键原则**：
- 原文不压缩，保留为可检索的知识库
- 风格理解是独立层，从原文"感受"而来
- 赛博作者可以随时回忆原文细节

## 输入/输出

### 输入
- 作者的大量文本（建议至少30万字）
- 每章一个文件，或整本合并文件
- 可选：作者笔名、作品列表

### 输出
- `knowledge_base/` - 知识库（原文片段+索引）
- `style_understanding.md` - 风格理解文档
- `cyber_author_prompt.md` - 赛博作者系统提示词
- `README.md` - 使用说明

## 流程

```
Phase 1: 知识库构建（拆分+索引原文）
Phase 2: 风格蒸馏（LLM阅读+感受+提炼）
Phase 3: 赛博作者构建（组装prompt+知识库接口）
Phase 4: 验证对话（和赛博作者对话测试）
```

---

## Phase 1: 知识库构建

### 目标
把原文拆成可检索的片段，建立索引，不丢失任何信息。

### 执行步骤

#### 1.1 拆分原文

按**场景/情节点**拆分，不是机械按字数：

```
拆分原则：
- 一个场景 = 一个片段（通常1000-5000字）
- 场景边界：时间跳转、地点转换、视角切换、章节分隔
- 保留完整上下文：每个片段包含"第X章"标签
- 极长章节可拆为多个场景，但要标注连续关系
```

输出格式：
```
knowledge_base/
├── chapters/
│   ├── ch001_scene01.md
│   ├── ch001_scene02.md
│   ├── ch002_scene01.md
│   └── ...
├── index.md          # 场景索引（章节号+场景描述+关键词）
└── metadata.json     # 元数据（总字数、章节数、场景数）
```

每个场景文件格式：
```markdown
# 第X章 - 场景N

[原文内容]

---
<!-- 元数据 -->
- 章节：第X章
- 场景：N
- 字数：XXXX
- 关键词：[角色A, 地点B, 事件C]
- 情节类型：[对话/描写/冲突/抒情]
```

#### 1.2 建立索引

索引要支持多种检索方式：
- 按章节号
- 按角色名
- 按情节类型（对话场景、描写场景、冲突场景等）
- 按关键词

Agent执行时，先快速浏览全文，建立章节-场景映射表。

## Phase 2: 风格蒸馏

### 目标
让LLM真正阅读大量原文，形成对作者风格的"感受"，不是统计分析。

### 执行方式

#### 2.1 批量阅读

```
阅读策略：
- 每批读10-20章（约5-10万字）
- 按顺序读，保持叙事连贯性
- 每批读完后记录风格观察
- 最后综合所有观察，形成整体理解
```

#### 2.2 风格观察记录

每批读完后，记录：

```
## 第N批阅读观察（第X章-第Y章）

### 这个作者的独特之处
1. [具体特征+原文例子]
2. [具体特征+原文例子]
3. ...

### 和其他作者的区别
- [区别点]

### 这个作者的"脾气"
- [作者的写作习惯、偏好、禁忌]
```

#### 2.3 综合理解

读完所有批次后，综合生成风格理解文档：

```markdown
# [作者名] 风格理解

## 核心标签（3-5个词，能区分这个作者）

## 叙事节奏
[作者的节奏习惯，附原文例子]

## 文字质感
[用词、句式、标点的偏好，附原文例子]

## 对话风格
[对话的特点，附原文例子]

## 描写手法
[描写的偏好，附原文例子]

## 情绪表达
[情绪处理方式，附原文例子]

## 叙事视角
[视角选择和距离感，附原文例子]

## 章节结构
[结构设计的规律，附原文例子]

## 这个作者的"脾气"
[写作习惯、偏好、禁忌]

## 最典型的3-5段原文
[最能体现这个作者风格的原文片段]
```

## Phase 3: 赛博作者构建

### 目标
组装一个可以直接对话的赛博作者，它能模仿写作、讨论技巧、分析原文。

### 3.1 系统提示词设计

```markdown
# 赛博作者：[作者名]

## 你是谁
你是[作者名]的赛博分身，拥有他的写作风格和创作理念。
你可以模仿他写作，也可以用他的视角讨论写作技巧。

## 你的写作风格
[插入风格理解文档的核心内容]

## 你的写作原则
1. [从原文中提炼的核心原则]
2. ...
3. ...

## 你的"脾气"
[写作习惯、偏好、禁忌]

## 你可以做的事
1. **模仿写作**：给出题材/大纲，你用[作者名]的风格写出来
2. **讨论技巧**：回答关于写作的问题，用[作者名]的视角
3. **分析原文**：分析[作者名]的作品，解释他的写作手法
4. **续写故事**：给出前文，你续写后续情节

## 你的知识库
你可以检索[作者名]的原文知识库，回忆具体细节。
当需要参考原文时，告诉用户你会检索知识库。

## 对话示例
用户：帮我写一个XXX场景
你：[用作者的风格写]

用户：这个作者怎么处理对话？
你：[分析+例子]

用户：这段原文为什么写得好？
你：[分析+引用原文]
```

### 3.2 知识库接口

赛博作者可以检索知识库，方式：
- 按章节号查
- 按角色名查
- 按情节类型查
- 按关键词查

Agent在对话时，根据需要检索相关场景片段。

### 3.3 输出文件

```
output/
├── knowledge_base/           # 知识库
│   ├── chapters/             # 场景片段
│   ├── index.md              # 索引
│   └── metadata.json         # 元数据
├── style_understanding.md    # 风格理解
├── cyber_author_prompt.md    # 赛博作者系统提示词
└── README.md                 # 使用说明
```

## Phase 4: 验证对话

和赛博作者对话，验证效果：

### 测试场景

1. **模仿写作测试**
   - 给一个新题材，让赛博作者写
   - 对比原文，看风格是否接近

2. **技巧讨论测试**
   - 问"你怎么处理对话？"
   - 看回答是否符合作者的风格特点

3. **原文分析测试**
   - 给一段原文，问"这段为什么写得好？"
   - 看分析是否准确

4. **续写测试**
   - 给一段前文，让赛博作者续写
   - 看续写是否保持风格一致

### 迭代优化

如果测试不满意：
- 补充更多原文阅读
- 调整风格理解的侧重点
- 优化系统提示词的指令

## Agent 执行流程

### 完整流程

```
用户：帮我蒸馏[作者名]的文风，创建赛博作者

Agent：
1. 确认输入目录
2. 扫描目录，统计总字数和章节数
3. Phase 1: 构建知识库
   - 拆分原文为场景片段
   - 建立索引
4. Phase 2: 风格蒸馏
   - 分批阅读原文（每批10-20章）
   - 记录每批的风格观察
   - 综合生成风格理解文档
5. Phase 3: 赛博作者构建
   - 组装系统提示词
   - 验证知识库可用性
6. Phase 4: 验证对话
   - 和赛博作者对话测试
   - 输出测试结果
7. 输出所有文件
```

### 快速模式

如果用户只想要系统提示词，不要赛博作者：
```
用户：生成[作者名]的风格提示词

Agent：
1. 跳过知识库构建
2. 只做风格蒸馏
3. 输出 style_prompt.md
```

### 增量模式

对于超大输入（1000万字+）：
```
1. 先处理前200万字，创建初版赛博作者
2. 和赛博作者对话，发现不足
3. 补充阅读后续文本，迭代优化
4. 重复直到满意
```

## 使用方式

### 方式1：完整流程
```
用户：蒸馏[作者名]的文风，创建赛博作者
```

### 方式2：只生成提示词
```
用户：生成模仿[作者名]风格的系统提示词
```

### 方式3：和赛博作者对话
```
用户：我想和[作者名]的赛博分身对话
Agent：[加载已有的赛博作者，直接进入对话]
```

### 方式4：分析特定方面
```
用户：分析[作者名]的对话风格
Agent：只聚焦对话维度分析
```

## 注意事项

1. **原文质量**：用作者原话，避免翻译或改编版本
2. **字数要求**：至少30万字，最好100万字以上
3. **阅读顺序**：按作品顺序阅读，保持叙事连贯性
4. **API成本**：大量阅读会消耗较多token，建议用flash模型
5. **迭代必要性**：一次分析可能不够准确，建议多次迭代
6. **知识库维护**：原文更新时，需要重建索引
